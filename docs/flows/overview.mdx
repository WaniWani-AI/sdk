A **flow** is a LangGraph-inspired state graph that compiles into a single MCP tool. The engine drives the conversation step by step: it asks questions, validates answers, branches on conditions, and persists state across tool calls — all from one declarative definition.

<Note>
  If you've ever tried to build a multi-turn MCP tool by hand — ask a question, return, hope the model asks the next one, handle state via raw string parsing — flows are the abstraction you wanted.
</Note>

## Why flows?

MCP tools are stateless. Every call starts from scratch. Building anything beyond a single-shot Q&A — onboarding, quoting, qualification, booking, support triage — usually means:

- Hand-rolling a state machine.
- Serializing state into tokens the model has to remember.
- Praying the model actually passes them back.
- Gluing together retry and resume logic.

Flows remove all of that. You define:

1. **State** — the fields you want to collect, as a Zod schema.
2. **Nodes** — handlers that either ask a question, run business logic, or show a widget.
3. **Edges** — the order in which nodes run, with optional branching.

You then `compile()` the graph into a `RegisteredFlow` and register it on your MCP server. The engine handles everything else.

## What you get for free

- **Automatic state persistence.** State lives server-side, keyed by the MCP session id — no round-tripping tokens through the model.
- **Step-by-step execution.** The engine advances one "step" per tool call: it runs action nodes, pauses at interrupts, resumes automatically on the next call.
- **Typed state.** State is inferred from your Zod schema, so `ctx.state` is fully typed inside every node.
- **Validation hooks.** Interrupt questions can include a `validate` function that runs after the user answers.
- **Resumability.** If the user wanders off and comes back, the flow picks up exactly where it left off.
- **Mermaid diagrams.** `flow.graph()` returns a ready-to-render Mermaid `flowchart TD` for documentation or debugging.

## A minimal flow

Here's a complete onboarding flow that asks for an email and a use case:

```ts
import { createFlow, END, START } from "@waniwani/sdk/mcp";
import { z } from "zod";

export const onboardingFlow = createFlow({
  id: "onboarding",
  title: "User Onboarding",
  description: "Use when a new user wants to get started.",
  state: {
    email: z.string().describe("Work email"),
    useCase: z.string().describe("What they want to build"),
  },
})
  .addNode("ask_email", async ({ interrupt }) =>
    interrupt({ email: { question: "What's your work email?" } }),
  )
  .addNode("ask_use_case", async ({ interrupt }) =>
    interrupt({
      useCase: {
        question: "What do you want to build?",
        suggestions: ["Analytics", "Support", "Lead capture"],
      },
    }),
  )
  .addEdge(START, "ask_email")
  .addEdge("ask_email", "ask_use_case")
  .addEdge("ask_use_case", END)
  .compile();
```

Register it on your server:

```ts
import { onboardingFlow } from "./journey/onboarding.js";

server.registerTool(
  onboardingFlow.name,
  onboardingFlow.config,
  onboardingFlow.handler,
);
```

That's it. The model will call `onboarding` with `{ action: "start", intent: "..." }`, receive an interrupt asking for the email, return it with `{ action: "continue", stateUpdates: { email: "..." } }`, and so on until the flow hits `END`.

## How execution works

Each tool call to a flow runs the engine until it either:

- **Pauses** at an interrupt or widget node, returning an `interrupt` or `widget` response to the model; or
- **Completes** by reaching `END`, returning a `complete` response.

Between calls, state is persisted server-side in a key-value store, keyed by the session id from `_meta`. The model never has to carry a token — when it calls the flow again on the same session, the engine loads the state automatically.

The engine auto-advances through **action nodes** (nodes that return plain state updates, not signals) without any round-trip to the model. Only `interrupt` and `showWidget` signals actually pause the flow.

## Tool contract

Flow tools accept a very small input shape:

```ts
type FlowToolInput =
  | { action: "start"; intent: string; stateUpdates?: Record<string, unknown> }
  | { action: "continue"; stateUpdates?: Record<string, unknown> };
```

- `start` — begins a new flow execution. `intent` is a short description of what the user is trying to do (used for dashboards and debugging).
- `continue` — resumes the current flow. `stateUpdates` carries the user's latest answer(s).

Flow tools return one of these shapes as their `content`:

| Status      | Meaning                                                                 |
| ----------- | ----------------------------------------------------------------------- |
| `interrupt` | Pause and ask one or more questions. Response includes `questions[]`.   |
| `widget`    | Pause and delegate rendering to a display tool.                         |
| `complete`  | Flow finished successfully.                                             |
| `error`     | Something went wrong. Response includes an `error` message.             |

You generally don't need to parse this yourself — the engine and the model handle the protocol. But it's useful to know when debugging.
