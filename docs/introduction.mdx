## What is WaniWani?

**WaniWani** is a platform for building, shipping, and measuring conversational MCP (Model Context Protocol) apps — the tools your AI assistant calls inside ChatGPT, Claude, and any other MCP-compatible host.

The **WaniWani SDK** (`@waniwani/sdk`) is the developer-facing library that plugs into your MCP server and gives you:

  <Card title="Event tracking" icon="chart-line" href="/tracking/overview">
    Automatic `tool.called` tracking, plus custom events, sessions, and user identification — all batched and shipped to the WaniWani dashboard.
  </Card>
  <Card title="Flows" icon="diagram-project" href="/flows/overview">
    LangGraph-inspired, multi-step conversational flows. Define a state graph, compile it into a single MCP tool, and let the engine drive the conversation.

## Why flows?

MCP tools are stateless by default. Every call starts from scratch, which makes it painful to build anything beyond a single-shot Q&A — quoting, onboarding, qualification, lead capture, booking, support triage.

A **flow** is a typed state machine that:

- Declares the data it needs to collect via a Zod schema.
- Pauses to ask the user questions (`interrupt`).
- Branches on conditions (`addConditionalEdge`).
- Persists state server-side, keyed by the MCP session.
- Resumes automatically on the next tool call — the model does not need to pass a token.

The result: multi-turn conversations that feel native to the assistant, without you having to handle state, retries, or correlation by hand.

## What this SDK is not

This SDK is **not** a replacement for your MCP server runtime. It works on top of the servers you already have — Skybridge, `@modelcontextprotocol/sdk`, `@vercel/mcp-handler`, and others. You keep control of your transport, auth, and deployment.

<Tip>
  If you already have a working MCP server (for example, an Alpic/Skybridge template), adding WaniWani takes a few lines: install the package, set an API key, and wrap your server with `withWaniwani()`.
</Tip>
