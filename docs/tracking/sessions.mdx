title: Sessions

A **session** in WaniWani represents one continuous conversation between a user and an MCP client (ChatGPT, Claude, etc.). Every event you track belongs to a session — that's how the dashboard groups events into funnels, counts unique users, and replays past conversations.

The important thing to know: **you almost never create a session manually**. The SDK derives it from the MCP request metadata that every tool handler receives as `extra._meta`.

## Why sessions come from `_meta`

Every MCP tool handler is called with two arguments:

```ts
async (input, extra) => {
  // extra._meta contains the host-provided correlation context
}
```

`extra._meta` carries the correlation identifiers needed to link a tool call back to a specific conversation. The SDK reads these identifiers and turns them into a stable session id. By passing `meta: extra._meta` into `client.track()`, you tell the SDK: _"this event belongs to the same conversation as the tool call that triggered it."_

## The golden rule

<Check>
  Always pass `meta: extra._meta` when tracking events from inside a tool handler.
</Check>

```ts
server.registerTool(
  "get_quote",
  { /* config */ },
  async ({ amount, currency }, extra) => {
    // ... do work

    await wani.track({
      event: "quote.succeeded",
      properties: { amount, currency },
      meta: extra._meta, // 👈 this is how sessions work
    });

    return { content: [{ type: "text", text: "Quoted!" }] };
  },
);
```

If you forget `meta`, the event still ships — but it arrives without a session id and can't be grouped with the rest of the conversation.

## Where `_meta` lives

Different MCP libraries expose request metadata in slightly different places. The SDK accepts all of them:

| Library                          | Where to find metadata           |
| -------------------------------- | -------------------------------- |
| `@modelcontextprotocol/sdk`      | `request.params._meta`           |
| `@vercel/mcp-handler`            | `extra._meta`                    |
| Skybridge (`skybridge/server`)   | `extra._meta`                    |

Anywhere you see `extra._meta` in the docs, substitute the equivalent for your runtime.

## Inside a flow

Flows handle session correlation for you. Each flow execution loads and saves its state keyed by the session id from `_meta`, so the user's progress survives across tool calls without the model having to pass any token. And the **session-scoped client** on `ctx.waniwani` automatically attaches `meta` for you:

```ts
.addNode("record", async ({ state, waniwani }) => {
  // No need to pass meta — the scoped client already has it
  await waniwani?.track({
    event: "quote.succeeded",
    properties: { amount: state.amount },
  });
  return {};
})
```

See [Flows overview](/flows/overview) for the full picture.

## Manual session ids

In rare cases — background jobs reacting to a finished session, retroactive data backfills, tests — you may need to set the session id explicitly. Do that with the `sessionId` field:

```ts
await wani.track({
  event: "purchase.completed",
  properties: { amount: 4999, currency: "USD" },
  sessionId: "session_01HX...",
});
```

<Warning>
  Only use manual `sessionId` as an escape hatch. Inside normal tool handlers, passing `meta: extra._meta` is simpler and less error-prone.
</Warning>

## Session lifecycle

The WaniWani platform generates `session.started` events automatically when it sees the first event carrying a new session id. You don't need to send `session.started` yourself.

Sessions don't have a hard "end" — they're considered closed after an idle timeout on the server side. If the user comes back later, they either resume the same session (if within the window) or start a fresh one, depending on how the host client re-connects.
