<Warning>
`toNextJsHandler`, `toExpressJsHandler`, and the underlying `createApiHandler` are **legacy APIs**. They are still exported for back-compat with existing customer MCPs but are no longer documented for new code.

**The chat widget no longer needs a self-hosted BFF.** [`ChatWidget`](/chat/react) and [`ChatEmbed`](/chat/embed) talk directly to `app.waniwani.ai` from the browser, with CORS enabled on the backend. The adapter routes — `/api/waniwani/[[...path]]` — are no longer required.
</Warning>

## What these adapters did

The adapters mounted a catch-all Next.js or Express route that did two unrelated things:

1. **`POST /api/waniwani`** — proxied chat requests to `app.waniwani.ai/api/chat` to keep the API key server-side
2. **`GET /api/waniwani/resource?uri=...`** — fetched MCP resource HTML for widget rendering

The pattern looked like this:

```ts
// app/api/waniwani/[[...path]]/route.ts
import { toNextJsHandler } from "@waniwani/sdk/next-js";
import { waniwani } from "@waniwani/sdk";

const wani = waniwani();

export const { GET, POST } = toNextJsHandler(wani, {
  chat: { mcpServerUrl: process.env.MCP_SERVER_URL! },
});
```

## Migration

If you only used the chat proxy:

```diff
- // app/api/waniwani/[[...path]]/route.ts deleted
+ // ChatWidget now talks directly to app.waniwani.ai — no route needed
```

Use `<ChatWidget projectId="wwp_..." />` directly. See [Chat in React](/chat/react).

If you used the resource fetcher (i.e. you also used legacy `createResource` widgets), see [Tools & resources (legacy)](/legacy/tools-resources) for the broader migration.

## If you're stuck on the legacy path

The adapters keep working — they're marked `@deprecated` but still functional. When the entry-point split happens you'll switch imports from `@waniwani/sdk/next-js` / `@waniwani/sdk/express-js` to `@waniwani/sdk/legacy/next-js` / `@waniwani/sdk/legacy/express-js`.
